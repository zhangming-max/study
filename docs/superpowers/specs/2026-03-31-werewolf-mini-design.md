# 狼人杀网页小游戏（单机主持人模式）UI 设计规格

## 1. 目标与范围
本项目实现一个可在本地离线运行的“狼人杀”网页小游戏，固定支持 **8 人**、角色为：
- `狼人` x2
- `预言家` x1
- `女巫` x1
- `守卫` x1`
- `平民` x3

采用 **单机本地主持人模式**：同一台设备上依次轮流点击操作，主持人（同一用户）在各阶段代替“群体角色的协同行为”完成选择；界面通过“屏幕遮罩/交接按钮”减少他人偷看身份与夜间信息的风险。

非目标（本期不做）：
- 多人联机/房间同步
- 扩展角色（如猎人、灵媒、丘比特等）
- 高阶规则（连守/连续预言等）除非作为可选开关且实现成本低

## 2. 用户故事（关键场景）
1. 作为主持人，我创建一局游戏，并让 8 位玩家依次查看自己的身份。
2. 到夜晚阶段时，我按顺序完成：守卫保护、狼人选择击杀、女巫处理、预言家查验。
3. 到白天阶段时，我公布昨夜死亡并让每位存活玩家完成投票。
4. 游戏结束后，我能看到胜利方，并可选择“复盘/重新开始”。

## 3. 规则与可配置项
### 3.1 固定规则（基础版）
- 每轮包含：夜晚（守卫 -> 狼人 -> 女巫 -> 预言家）-> 白天（公布 -> 投票）-> 结算。
- 守卫、预言家、女巫只能选择 **存活玩家**（默认规则：不允许对已死亡/出局者操作）。
- 投票：每位存活玩家投 1 票。
- 平票处理（默认）：平票无人出局，直接进入下一轮夜晚。

### 3.2 可选开关（基础版可做，但不强制）
- `平票重投一次`：若投票平票，则只重投一次；若仍平票则无人出局。
- `守卫是否允许连守同一人`：默认不允许（实现上通过记录上一轮守护目标进行限制）。

## 4. 页面与交互流程
采用单页应用（SPA）结构：一个入口，多个阶段屏幕切换。每个屏幕包含明确的阶段标题、操作说明、阶段进度提示。

### 4.1 入口页：开始与配置
- 显示固定人数（8）与角色构成说明。
- 主操作按钮：
  - `开始发牌`
  - `重新开始（清空旧局）`
- 可选：展示“将以主持人模式进行夜间行动选择”的说明。

### 4.2 身份查看页：屏幕交接
- 8 个座位从 1 到 8。
- 按座位顺序显示“请交给玩家 X 查看身份”。
- 操作方式：
  - 进入该座位后弹出遮罩层 + 身份 reveal（可通过倒计时/确认按钮隐藏身份）。
  - 主按钮：`交给下一位`（离开该座位时立即隐藏身份信息）。
- 身份展示完成后自动进入夜晚的第一阶段。

### 4.3 夜晚阶段（按顺序四步）
夜晚阶段界面保持一致的布局：左侧阶段信息/说明，右侧选择目标（或确认结果）。

1. `守卫行动`
   - 主按钮：`选择守护目标`
   - 目标列表仅包含存活玩家
   - 允许在该步内 `清除/重新选择`
2. `狼人行动`
   - 主按钮：`狼人选择击杀对象`
   - 目标列表仅包含存活玩家，且允许选择“无击杀”作为容错（默认禁用，可选开关）
   - 选择完成后进入女巫阶段，确保非女巫步骤不暴露该信息（仅对主持人操作视角显示“已记录狼人选择”）
3. `女巫行动`
   - 若昨夜存在狼人击杀对象（且未被守卫成功守护），才显示：
     - `是否使用解药（一次）`
     - 若解药不使用或不符合条件：展示 `是否使用毒药（一次）`
   - 毒药/解药各最多一次，使用后对应按钮禁用并提示剩余次数。
4. `预言家行动`
   - 仅对存活玩家展示可选名单
   - 主按钮：`查验目标`
   - 查验结果对主持人短暂可见（离开该步骤立即清空结果显示）

### 4.4 白天阶段
1. `公布死亡`
   - 列表展示昨夜最终死亡的玩家（显示“出局/死亡”）
   - 若无人死亡，显示 `平安夜`
2. `白天投票`
   - 给每位存活玩家显示 `投票对象`选择（主持人依次点击“确认该玩家投票”）
   - 每投一次更新计票，但前端不展示真实身份，仅展示投票过程与当前计票。
   - 投票结束后：
     - 若存在最高票单一者：该玩家出局
     - 若平票：按“默认无人出局”或“可选平票重投一次”

### 4.5 结算页
- 显示胜利方与获胜条件：
  - `狼人全灭 -> 好人胜`
  - `狼人数量 >= 好人数量 -> 狼人胜`
- 提供按钮：
  - `开始新局`
  - `复盘（查看时间线）`

### 4.6 复盘（时间线）
- 显示按轮次的简化时间线：
  - 夜晚：守卫目标、狼人击杀目标（或无击杀）、女巫是否使用解药/毒药、预言家查验结果（仅主持人视角）
  - 白天：死亡名单、投票结果、出局者
- 提供 `返回结算` 与 `清空复盘`。

## 5. 状态机设计（Phase + Turn）
### 5.1 全局状态
`GameState`：
- `players: Player[]`（8 个）
- `turn: number`（从 1 开始）
- `phase: Phase`（当前阶段）
- `phaseIndex`（夜晚四步/白天两步的内部序号）
- `night: NightState`（本轮夜晚临时记录）
- `history: HistoryEvent[]`（复盘数据）
- `settings: Settings`（平票重投、连守限制等）
- `revealProgress`（身份查看到第几个座位）

### 5.2 Phase 枚举（建议）
- `setup`
- `reveal`
- `night.guard`
- `night.wolf`
- `night.witch`
- `night.seer`
- `day.announce`
- `day.vote`
- `end`

### 5.3 转移规则（概要）
- `setup -> reveal`：生成并打乱身份；初始化 players alive 状态为 true
- `reveal -> night.guard`：所有玩家身份完成展示后进入第一夜
- `night.guard -> night.wolf`：记录守卫目标（若允许）
- `night.wolf -> night.witch`：记录狼人击杀目标；计算“是否被守卫守住（本轮）”
- `night.witch -> night.seer`：根据狼人击杀是否成立决定是否进入女巫可选操作
- `night.seer -> day.announce`：记录查验目标与结果（仅主持人界面短暂显示）
- `day.announce -> day.vote`：更新 players alive 状态，准备投票
- `day.vote -> end | night.guard`：
  - 若满足胜利条件：进入 `end`
  - 否则：turn++，进入下一轮 `night.guard`

## 6. 组件结构（前端拆分建议）
- `AppShell`：顶部标题/全局提示；统一按钮样式与进度条
- `StartScreen`
- `RevealScreen`
- `NightGuardStep`
- `NightWolfStep`
- `NightWitchStep`
- `NightSeerStep`
- `DayAnnounceStep`
- `DayVoteStep`
- `EndScreen`
- `ReplayTimeline`
- `ConfirmModal`：危险操作（重开/清空）确认
- `MaskedReveal`：身份遮罩/屏幕交接通用组件
- `PlayerList`：座位/存活标记组件（统一渲染规则）

## 7. 数据结构草案
`Player`：
- `seat: number`（1..8）
- `name?: string`（可选：可不做）
- `role: Role`
- `alive: boolean`
- `voteCount: number`（白天阶段临时统计，可派生或存储）

`NightState`：
- `guardTargetSeat?: number`
- `wolfTargetSeat?: number`
- `witchSaved?: boolean`（是否使用解药挽救）
- `witchPoisonTargetSeat?: number`
- `seerTargetSeat?: number`
- `seerResult?: "狼人" | "好人"`

`HistoryEvent`：
- `turn: number`
- `type: "night_guard" | "night_wolf" | ... | "day_vote" | ...`
- `payload: object`

## 8. 错误处理与边界条件
- 操作时若发现“当前阶段不允许该操作”（例如投票阶段点击夜晚按钮），直接禁用并提示。
- 所有选择列表基于 `alive` 派生，避免出现对已死亡目标操作。
- 当女巫解药/毒药次数用完，按钮禁用并显示剩余次数提示。
- 在“重开/清空”时使用确认弹窗，防误触。

## 9. 可访问性（A11y）与交互质量
依据 UI/UX Pro Max 规则：
- 所有可点击元素提供可见 `focus` 状态。
- 图标按钮必须有 `aria-label`；不使用纯图标承载信息。
- 保证触控目标尺寸接近 `44x44px`（移动端视口下）。
- 重要信息（身份、投票结果、死亡公布）使用足够对比度文字颜色。
- 盲操作避免依赖颜色：如使用图标/文字同时表达状态。
- 遵循 `prefers-reduced-motion`：动画减少/关闭。

## 10. 手工测试计划（交付前验证）
1. 从开始页到第 1 夜，确认身份遮罩在每次“交给下一位”后立即隐藏。
2. 夜晚四步依次走完：守卫守住/未守住、女巫解药挽救、女巫毒药击杀路径。
3. 白天投票：单一高票出局、平票无人出局、（可选）平票重投一次。
4. 结算条件：狼人全灭 vs 狼人数量 >= 好人数量。
5. 页面刷新：若实现本地保存/载入，确认状态恢复正确且阶段不乱。

## 11. 实施注意事项（与后续代码对齐）
- 优先把“状态机”和“可见信息范围”做清楚：每个阶段只暴露该阶段允许的信息。
- 复盘数据以 `history` 统一记录，不要从 UI 反推。
- 即使单机，也要把“主持人操作”和“玩家交接”概念抽象为可复用 UI。
